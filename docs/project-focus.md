## Current Focus Areas

- Deliver Manus-style operator workspace parity on the deployed-agent chat surface at `http://localhost:3000/agents/<agent_id>/chat`, centered on `agent-builder-ui/app/(platform)/agents/[id]/chat/page.tsx` and `agent-builder-ui/app/(platform)/agents/[id]/chat/_components/TabChat.tsx`.
- Expand the current `Agent's Workspace` from `terminal` + `thinking` only into a multi-surface workspace that supports browser use, code use, editor use, terminal use, artifact previews, and connected-tool workflows without leaving the product.
- Translate the parity goal into feature packages that leave a human-testable improvement on the deployed-agent chat page after each worker run, rather than backend-only groundwork with no visible operator win.

## Desired Outcomes

- Browser use parity:
  live cloud-browser session or browser replay inside the chat page, local-browser/operator mode for real logged-in sessions when needed, auth/takeover handoff UX, and browser history/action visibility.
- Code use and editor parity:
  file tree, file viewer/editor, diffs, code copy/export/download, side-by-side preview, and an explicit path from agent output to code ownership rather than opaque chat-only generation.
- Terminal and computer-use parity:
  richer terminal output with command history, process state, stdout/stderr, deep links into touched files, retry/stop controls, and a roadmap toward broader desktop/computer-use actions where the security model allows it.
- Artifact parity:
  first-class workspace support for generated reports, slides, dashboards, webpages, images, and downloadable code/assets, with preview and export surfaces instead of hiding outputs in chat prose.
- Research and connected-tool parity:
  MCP/connectors-aware workflows, real external context in agent tasks, and a path toward Wide Research-style multi-source research outputs that land in the same workspace.
- Productization parity:
  publish/redeploy flows, app preview, access control/auth, analytics, data/database visibility, and other "turn output into a real product" capabilities on the same operator journey.
- Persistence parity:
  project-level memory, reusable instructions/files, and task continuity so deployed agents can behave like durable workspaces rather than one-off chats.

## In-Scope Requirement Signals

- Current surface gap:
  `TabChat.tsx` currently hardcodes `useState<"terminal" | "thinking">("terminal")` and only renders `terminal` plus `thinking` tabs in the `Agent's Workspace`. Treat the missing browser, files/editor, artifact, and richer tool-state surfaces as an explicit parity gap, not as a vague future nice-to-have.
- Browser-focused feature packages:
  browser panel, browser action timeline, DOM/screenshot/video replay, cloud-vs-local browser mode, take-over/resume flow, auth/session prompts, and browser-state persistence.
- Code/editor-focused feature packages:
  file tree, file open-from-terminal, multi-file editor, diff/revision history, code-control export/download, GitHub handoff, and live preview coupling.
- Terminal/computer-use feature packages:
  structured tool-call rendering, process lifecycle, task-level command grouping, terminal-to-file navigation, safe interruption, and any secure expansion beyond shell-only work.
- Artifact-focused feature packages:
  artifact registry/gallery, preview panels for slides/reports/dashboards/webpages, export/share actions, and durable metadata so outputs survive page refreshes and handoffs.
- Connectors/research-focused feature packages:
  connected-tool state, connector configuration visibility from the chat workspace, research-mode outputs, and workflows that combine browser actions with external data sources.
- Productization-focused feature packages:
  deploy/publish status, app preview, auth/access control, analytics, data layer visibility, and settings/mission-control surfaces needed to make generated apps operable.
- Cross-layer work is allowed when it directly serves this parity focus:
  backend APIs, SSE/tool events, sandbox/browser lifecycle, artifact persistence, deploy metadata, and connector plumbing are in scope when they unlock a visible chat-workspace capability.
- Suggested delivery order:
  1. Browser visibility + takeover
  2. Files/editor + artifact preview
  3. Richer terminal/process state + file navigation
  4. Connector-aware workflows + research outputs
  5. Publish/auth/analytics/data operator surfaces
  6. Persistent project/workspace memory polish

## Explicit Non-Goals

- Do not optimize for pixel-perfect Manus visual cloning. Match capability and operator flow, not brand or exact UI chrome.
- Do not spend focus cycles on unrelated marketing pages or generic dashboard polish outside the deployed-agent chat/workspace journey unless required by the parity feature.
- Do not collapse the parity effort into one giant all-or-nothing task. The analyst should emit sequenced feature packages that each leave something a human can test on the target page.
- Do not introduce unsafe privileged local-computer control without an explicit approval, isolation, and security model.

  Manus documents direct code copy plus full codebase download/export so the user retains ownership and portability.

  Manus documents project-level master instructions and attached knowledge/files for recurring work.

- Wide research and non-chat outputs:
  Manus documents Wide Research plus data-analysis/visualization outputs such as slide decks, dashboards, reports, and standalone webpages.
