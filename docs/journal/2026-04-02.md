## Flutter browser-targeted repro and fix

- Reproduced the `ruh_app` browser issue with real browser inspection instead of only terminal logs.
- Findings:
  - `flutter test integration_test/login_flow_test.dart -d chrome` is not a valid path here because Flutter does not support web devices for integration tests in this setup.
  - `flutter run -d web-server --profile` fails immediately because `ruh_app` has no checked-in `web/` directory, so browser runs are currently a debug-only inspection path.
  - The debug web startup path could throw `_RenderTheater` layout assertions when the debug-only `DebugOverlay` wrapped the router output, so web debug runs now skip that overlay.
  - The real user-visible browser failure was backend CORS, not Flutter auth logic: `ruh-backend` did not allow `http://localhost:4001`, so the browser login/bootstrap flow showed a transport error until the Flutter web dev origins were added to `ALLOWED_ORIGINS`.
- Updated `ruh-backend/.env` and `ruh-backend/.env.example` to include `4000`/`4001` localhost origins, added a browser startup smoke script under `ruh_app/test/browser_startup_smoke.py`, and documented the debug-only browser caveat in `docs/knowledge-base/018-ruh-app.md`.

- Added regression coverage proving mixed developer+customer users do **not** advertise `customer` access until the active org is switched to customer.

## Flutter runtime contract repair for chat, browser, and files

- Closed the remaining Flutter customer-runtime gaps after the auth fix by tracing the live seeded sandbox instead of guessing from UI symptoms.
- Root cause summary:
  - `GET /api/sandboxes/:sandbox_id/status` was still proxying the nonexistent gateway path `/api/status` instead of the real OpenClaw health path `/health`, so the app could see a running container but still mark the runtime unhealthy.
  - the browser stack binaries were installed in the sandbox, but `Xvfb`, `x11vnc`, and `websockify` were not always running after launch/restart, so browser status and screenshot routes could fail until an operator manually restarted those services.
  - the OpenClaw operator WebSocket did not emit live tool frames for ordinary runs even when the transcript clearly contained `toolCall` + `toolResult` messages, so the Flutter terminal/browser panels never saw tool activity.
  - the Flutter browser panel was still using a generic JSON-style GET path for `/browser/screenshot`, which is unreliable for raw JPEG bytes.
- Implemented the backend/runtime fixes:
  - added `ruh-backend/src/sessionToolTranscript.ts` to resolve the current session transcript and extract only the latest turn's `tool_start` / `tool_end` events from `toolCall` and `toolResult` entries
  - updated `ruh-backend/src/app.ts` so `/api/sandboxes/:sandbox_id/chat/ws` replays those structured tool events before `[DONE]` whenever the live operator socket does not surface them
  - changed sandbox status probing to gateway `/health`
  - made `/api/sandboxes/:sandbox_id/browser/status` and `/browser/screenshot` self-heal the interactive browser stack by re-running `ensureInteractiveRuntimeServices()` and retrying once before returning inactive/fallback output
- Implemented the Flutter/client fixes:
  - added `BackendClient.getBytes()` in `ruh_app/lib/services/api_client.dart`
  - updated `BrowserPanel` to fetch raw screenshot bytes through that path and added a widget regression for it
  - updated `HealthMonitor` to derive health from `SandboxHealth.fromJson()` so `gateway_reachable` drives the result instead of brittle string matching
- Updated/expanded regression coverage:
  - `ruh-backend/tests/unit/sessionToolTranscript.test.ts`
  - `ruh-backend/tests/e2e/chatProxy.test.ts`
  - `ruh_app/test/widgets/browser_panel_test.dart`
  - plus the existing focused Flutter service tests and backend launch/runtime tests

## Flutter bearer-session continuity fix

- Reproduced the reported native login problem against the live macOS desktop integration test instead of relying on the smaller auth unit harness.
- Root cause summary:
  - the Flutter app depended on immediately reading tokens back from platform storage for the next authenticated request after login or refresh
  - backend `GET /api/auth/me` rebuilt active-org context from the refresh-token cookie only, which is valid for browser clients but wrong for native bearer-token clients
  - the existing desktop integration test had also drifted behind the current customer shell by still waiting for the old builder-only `Create Agent` control
- Implemented the auth/session continuity fixes:
  - `ruh_app/lib/services/access_token_store.dart` now keeps access + refresh tokens cached in-process so the next authenticated request does not depend on secure-storage readback
  - `ruh_app/lib/services/auth_service.dart` now restores the locally stored refresh token into `AuthSession` during bootstrap
  - `ruh-backend/src/authRoutes.ts` now lets `GET /api/auth/me` fall back to the authenticated bearer token's `orgId` when no refresh-token cookie exists
  - `ruh_app/integration_test/login_flow_test.dart` now clears both stored tokens before startup and asserts against the current authenticated customer shell (`Workspace`)
