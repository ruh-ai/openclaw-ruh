# Openclaw Ruh Workspace

Use this skill to stay aligned with the repository's project model, source-of-truth rules, and operational tooling.

Read `references/workflows.md` for the concrete repo map, commands, and task checklists.

## Start Here

- Work from the repository root.
- Read `docs/Knowledge Base/Project Knowledge/Project Knowledge Hub.md` first when you need project context quickly.
- Read `docs/01 Sources/README.md` when you need the raw source library.
- Read `docs/02 Operations/README.md` and `docs/02 Operations/Linear/README.md` when the task affects planning, backlog structure, or Linear.

## Understand The Project

1. Read `docs/Knowledge Base/Project Knowledge/Project Brief.md` to anchor on the product thesis.
2. Read `docs/Knowledge Base/Project Knowledge/Architecture Model.md`, `docs/Knowledge Base/Project Knowledge/Packs and Overlays.md`, and `docs/Knowledge Base/Project Knowledge/Delivery Model.md` to understand the system shape.
3. Read `docs/Knowledge Base/Documents/` when you need source-backed summaries, metadata, and cross-links.
4. Open the raw files under `docs/01 Sources/` when the exact wording of a PDF, DOCX, or overlay manifest matters.

## Change Documents Safely

1. Treat `docs/01 Sources/` as the source of truth for imported source artifacts.
2. Treat `docs/02 Operations/` as the home for execution plans, backlog guidance, and operational notes.
3. Treat any note carrying `<!-- Generated by scripts/sync_obsidian_kb.py. Do not hand-edit. -->` as generated output.
4. Edit source or curated notes instead of generated notes, then regenerate the knowledge base.
5. Keep source edits and regenerated output in the same change.

## Change Linear And Planning Safely

1. Use the repo-local CLI via `npm run linear -- ...` instead of a separate MCP path.
2. Verify auth and read the current state before making writes.
3. Re-read the affected project, milestone, label, or issue objects after every write batch.
4. Update `docs/02 Operations/Build Plan.md` in the same session after material planning changes in Linear.
5. Update `docs/02 Operations/Linear/README.md` when the Linear operating model changes.
6. Use `scripts/import_linear_backlog.py` for backlog seeding or bulk bootstrap, not for routine day-to-day issue edits.

## Preserve The Project Model

- Keep the reusable unit as the employee pack, not a customer-specific prompt bundle.
- Keep customer-specific mappings in tenant overlays, not in shared pack content.
- Preserve the framing that OpenClaw provides the runtime layer and Ruh.ai adds the control plane, pack system, approvals, evals, and analytics.
- Remember the first target domains are AI BuildOps and Construction Project Operations.
- Keep the design-partner pilot model in mind: bounded workflows, approvals, and measurable outcomes before wider autonomy.

## Validate The Result

- Run `python3 scripts/sync_obsidian_kb.py` whenever `docs/` changes.
- Validate backlog-import changes in dry-run mode before proposing workflow changes.
- Keep generated files and source edits in the same pull request.
- Update `README.md` or `CONTRIBUTING.md` when contributor workflow changes.
- Avoid committing secrets, API tokens, local workspace state, or machine-specific files.
