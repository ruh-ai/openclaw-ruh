# 2026-04-06

## Codex

- Investigated a local `agent-builder-ui` failure on `http://localhost:3000` where the UI stayed on the loader/spinner.
- Reproduced the issue with Playwright from `agent-builder-ui` and captured the browser failure mode: required `/_next/static/*` assets were returning HTML `404` fallback responses instead of JS/CSS. The broken requests included `/_next/static/chunks/main-app.js`, `/_next/static/chunks/app-pages-internals.js`, `/_next/static/chunks/app/(auth)/authenticate/page.js`, and `/_next/static/css/app/layout.css`.
- Confirmed the running dev server had an inconsistent `.next` state: the manifests referenced app chunks that were missing under `.next/static`, which prevented hydration and left the page on the server-rendered loader.
- Restarted the builder dev server with `cd agent-builder-ui && bun run dev`. After restart, the same asset URLs returned `200`, and Playwright verified the local auth page rendered correctly at `/authenticate?redirect_url=%2F`.
- No application code changes were required. Durable takeaway: when the builder shows only the spinner, check `/_next/static/*` asset responses before tracing auth/bootstrap logic, because a stale Next dev server can mimic an auth hang.
