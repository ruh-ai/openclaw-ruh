# LEARNING: Operator-defined runtime inputs are documented but not implemented

[[000-INDEX|← Index]] | [[013-agent-learning-system]] | [[SPEC-agent-learning-and-journal]]

## Context

`Analyst-1` reviewed the active Google Ads creation-focus lane on 2026-03-27 to identify one non-duplicate missing feature package. The KB and API docs already described `runtimeInputs[].source = operator_defined`, so the expectation was that operators could add extra non-secret runtime values when the architect or skill graph did not derive them.

## What Was Learned

The documented `operator_defined` branch of the runtime-input contract does not exist in the live codebase yet. The current frontend and backend only accept `architect_requirement` and `skill_requirement`, and the shared Runtime Inputs editor is list-only, so operators cannot add a custom non-secret runtime input even though the KB says they can.

## Evidence

- At the time of the original audit, [[SPEC-google-ads-agent-creation-loop]] and [[004-api-reference]] both said `runtimeInputs[].source` may be `operator_defined`.
- [`/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/lib/agents/types.ts`](/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/lib/agents/types.ts) defines `AgentRuntimeInputSource` as only `architect_requirement | skill_requirement`.
- [`/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/ruh-backend/src/validation.ts`](/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/ruh-backend/src/validation.ts) validates only those two source values for persisted runtime inputs.
- [`/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/ruh-backend/src/agentStore.ts`](/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/ruh-backend/src/agentStore.ts) normalizes any other runtime-input source back to `architect_requirement`.
- [`/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/app/(platform)/agents/create/_components/configure/StepRuntimeInputs.tsx`](/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/app/(platform)/agents/create/_components/configure/StepRuntimeInputs.tsx) only edits existing rows; it offers no add/remove path for operator-created inputs.
- [`/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/lib/agents/runtime-inputs.ts`](/Users/prasanjitdey/Documents/workspace/work/projects/openclaw-ruh-enterprise/agent-builder-ui/lib/agents/runtime-inputs.ts) already preserves extra existing inputs whose keys are not derived from the current graph, which means the missing piece is primarily contract/UI parity rather than a missing merge primitive.

## Implications For Future Agents

- Treat the KB/API `operator_defined` source as intended behavior, not shipped behavior, until the runtime-input contract and editor are extended together.
- When working on Google Ads create-flow truthfulness, do not assume every required non-secret runtime value can be generated by the architect. The operator may need a first-class custom-input path.
- Keep this work separate from secret handling. Custom runtime inputs belong in `runtimeInputs[]` and `~/.openclaw/.env`, while secret material must remain behind the credential routes.
