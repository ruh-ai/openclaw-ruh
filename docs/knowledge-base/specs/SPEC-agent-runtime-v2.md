## Summary

Every agent gets its own full-stack application: a SQLite database for structured data, a vector store for RAG and memory, a custom API layer for exposing data, and a Mission Control dashboard for end users to see what the agent is doing. The architect generates all of this during the Think → Plan → Build flow based on the agent's purpose.

## The Vision

**Before:** An agent is SOUL.md + skills + triggers. It can DO things but can't REMEMBER, STORE, or SHOW anything.

**After:** An agent is a full-stack application with a soul:

```
Container (sandbox)
├── OpenClaw gateway         (chat API — existing)
├── SOUL.md + skills/        (personality + actions — existing)
├── agent-runtime/           (NEW: custom backend + dashboard)
│   ├── SQLite database      (structured data: tables the agent populates)
│   ├── Vector store         (RAG: embeddings for memory and retrieval)
│   ├── API routes           (custom endpoints for the agent's data)
│   └── Dashboard pages      (Mission Control: what end users see)
└── triggers/                (when the agent acts — existing)
```

## Architecture

### Stack Choice

| Component | Technology | Why |
|-----------|-----------|-----|
| Database | SQLite (better-sqlite3) | Already in sandbox image. Zero-config, WAL mode, fast |
| Vector store | SQLite + float32 arrays | Same DB, store embeddings as BLOBs, cosine similarity in JS |
| API server | Next.js API routes | Already in agent-runtime, hot-reload, TypeScript |
| Dashboard | Next.js React pages | Already in agent-runtime, server components, Tailwind |
| Charts | Lightweight (recharts or Chart.js) | Agent generates data configs, components render them |

### What the Architect Generates

During the creation flow, the architect produces these NEW artifacts:

1. **`runtime/schema.sql`** — CREATE TABLE statements for the agent's domain data
2. **`runtime/seed.sql`** — Optional initial data
3. **`runtime/routes/*.ts`** — Custom API route files (Next.js API route convention)
4. **`runtime/pages/*.tsx`** — Custom dashboard page files
5. **`runtime/vectors.json`** — Vector collection definitions (what to embed, what fields)

These get written to the agent's workspace alongside SOUL.md and skills/.

### Auto-Migration System

The agent-runtime's `db-init.mjs` is extended to:
1. Always create the core tables (tasks, reports, activity_log, calendar_events)
2. Auto-discover and run `workspace/runtime/schema.sql` if it exists
3. Store migration state so re-running is safe (track applied schemas by hash)

### Vector Store Design

Embeddings stored in SQLite:

```sql
CREATE TABLE IF NOT EXISTS vectors (
  id TEXT PRIMARY KEY,
  collection TEXT NOT NULL,      -- e.g. "memory", "documents", "knowledge"
  content TEXT NOT NULL,          -- original text
  embedding BLOB NOT NULL,        -- float32 array as binary
  metadata TEXT DEFAULT '{}',     -- JSON metadata
  created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_vectors_collection ON vectors(collection);
```

Vector search is done in JavaScript:
```typescript
function cosineSimilarity(a: Float32Array, b: Float32Array): number { ... }
function search(collection: string, queryEmbedding: Float32Array, topK: number): Result[] { ... }
```

Embedding generation: the agent calls an embedding API (OpenAI, Anthropic, or local) through a skill. The vector store just stores and retrieves.

### How Skills Use the Database

Skills can import the db module and read/write directly:

```typescript
// skills/daily-report/SKILL.md references this
// The skill's implementation runs inside the container
import { getDb } from "../../agent-runtime/src/db";

const db = getDb();
db.prepare("INSERT INTO campaign_metrics (date, impressions, clicks, cost) VALUES (?, ?, ?, ?)")
  .run(today, impressions, clicks, cost);
```

### How the Dashboard Shows Data

The dashboard pages are standard Next.js pages that fetch from API routes:

```tsx
// runtime/pages/campaigns.tsx (generated by architect)
export default function CampaignsPage() {
  const { data } = useFetch('/api/campaigns/stats');
  return (
    <DashboardLayout title="Campaign Performance">
      <MetricCards metrics={data?.summary} />
      <LineChart data={data?.daily} xKey="date" yKey="impressions" />
      <DataTable columns={['Campaign', 'Status', 'Spend', 'ROAS']} rows={data?.campaigns} />
    </DashboardLayout>
  );
}
```

## Changes to Architect Instructions

### THINK_SYSTEM_INSTRUCTION additions

The PRD must now include:
- **Data Requirements**: What data does this agent collect, process, and store?
- **Dashboard Requirements**: What does the end user need to see? What metrics, tables, charts?
- **Memory Requirements**: What context does the agent need to remember across conversations?

The TRD must now include:
- **Data Schema**: Table definitions with columns and types
- **API Endpoints**: What data to expose and how
- **Dashboard Pages**: Page layouts with component types
- **Vector Collections**: What to embed and when

## Component Library

The dashboard has a pre-built component library that the architect uses:

| Component | Props | Use |
|-----------|-------|-----|
| `<MetricCards>` | `metrics: { label, value, trend?, icon? }[]` | KPI summary |
| `<DataTable>` | `columns, rows, sortable?, searchable?` | Tabular data |
| `<LineChart>` | `data, xKey, yKeys, colors?` | Time series |
| `<BarChart>` | `data, xKey, yKey, color?` | Comparisons |
| `<PieChart>` | `data, nameKey, valueKey` | Distribution |
| `<StatusBadge>` | `status, label?` | Status indicators |
| `<ActivityFeed>` | `items: { icon, text, time }[]` | Event timeline |
| `<EmptyState>` | `title, description, action?` | No-data fallback |
