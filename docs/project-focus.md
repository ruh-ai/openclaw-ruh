## Vision

**Ruh.ai is the place where enterprises create digital employees with a soul.**

Not bots. Not automations. Assistants you love to work with — who understand you, remember you, and feel like real teammates. They have personality, context, and judgment. They grow with you.

## Mission

Build the platform that lets any enterprise create, deploy, and manage AI assistants that feel human — starting with a powerful end-to-end creation experience and a client application where people actually work alongside their digital employees every day.

## Product Shape

- **Agent Builder** (`agent-builder-ui`) — where you create and shape your assistant's soul: personality, skills, tools, triggers, memory.
- **Client Application** (`ruh-frontend`) — where end users work with their assistants daily. Direction: may become a desktop application.
- **Backend Infrastructure** (`ruh-backend`) — sandbox orchestration, agent lifecycle, persistence, deployment.

## Proving Case: Google Ads Agent

The **Google Ads agent** is the first assistant we are building on the platform. Every creation-flow feature, configuration step, deployment path, and improvement loop is validated against this single agent. When we say "create an agent," we mean the Google Ads agent.

This agent should be:
- A credible, domain-specific ads management assistant — not a generic demo bot
- Built end-to-end through the builder: create, review, configure tools/triggers, save, deploy, improve
- The concrete acceptance benchmark: if a feature doesn't make the Google Ads agent better or easier to build, it's not the priority

## Current Focus Areas

- **End-to-end Google Ads agent creation:** The full builder journey — from first conversation through configure, review, save, deploy, and improve — should produce a real, working Google Ads assistant.
- **MCP-first tool configuration:** Tool setup must be driven by real connector state, safe credential handling, and runtime-backed configuration. No more mock cards or decorative selections.
- **Agent personality and memory:** The creation experience should feel like you're shaping a colleague — their voice, style, domain knowledge, and how they remember and adapt. This is what makes a "digital employee with a soul."
- **Persisted configuration that matters:** Choices made in Connect Tools, Choose Skills, and Set Triggers must shape the actual persisted agent record and runtime behavior. No decorative config that drops data on save.
- **Iterative self-improvement:** The builder should surface concrete improvements while the Google Ads agent is being created. Approved improvements feed back into persisted agent config, not just chat prose.
- **Client application direction:** Begin evaluating ruh-frontend as the daily-driver surface where users work alongside their assistants. Consider desktop app conversion.

## Desired Outcomes

- A human can create the Google Ads agent from `/agents/create`, shape its personality and domain expertise, configure its tools and triggers, save it, deploy it, and return later through Improve Agent without losing anything important.
- The Google Ads agent feels like a knowledgeable ads specialist you'd want on your team — not a generic chatbot wearing a Google Ads skin.
- The Configure flow uses real MCP-oriented connection state, including connector discovery, configured/not-configured status, safe secret handling, and only the options that actually work at runtime.
- The creation loop itself improves as the agent is built: the builder proposes better next steps, missing config, or workflow upgrades, and the operator can accept those into saved agent state.
- The client application surface is identified and initial work begins on making it the place where users interact with their deployed assistants daily.

## Suggested Delivery Order

1. Google Ads agent creation flow — end-to-end through the builder with real config
2. MCP-first tool connections for the Google Ads lane
3. Agent personality, voice, and memory configuration in the builder
4. Runtime-backed triggers and deploy/apply path
5. Builder-surfaced process improvements accepted into persisted config
6. Client application evaluation and initial deployed-agent experience
7. Improve Agent polish and regression coverage for save/reopen/redeploy

## Explicit Non-Goals

- Do not broaden into a generic connector marketplace or all-domains template system before the Google Ads vertical slice works end-to-end.
- Do not ship decorative configuration that drops data on save, hides unsupported runtime gaps, or stores secrets unsafely.
- Do not treat chat-only commentary as enough. If the builder suggests an improvement, the accepted version must become visible persisted product state.
- Do not optimize the client application experience before the creation experience produces a real, working agent.
